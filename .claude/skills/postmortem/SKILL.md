# /postmortem — バグ分析レポート生成

## 概要
バグ発生時に5軸で構造化分析を行い、Notionに登録可能なレポートを出力する。

## 起動条件
ユーザーが `/postmortem` と入力したとき、またはバグ分析を依頼されたとき。

## 入力
- バグの概要（何が起きたか）
- 再現手順（わかる範囲で）
- 影響範囲

## 分析5軸

### 1. 技術原因（Technical Root Cause）
- 直接的なコードレベルの原因
- 該当ファイル・行番号・関数名を特定

### 2. 構造原因（Structural Cause）
- アーキテクチャ・設計上の問題
- 責務分離の不備、依存関係の問題など

### 3. プロセス原因（Process Cause）
- 開発フロー上の問題
- レビュー漏れ、テスト不足、デプロイ手順の問題など

### 4. 認知原因（Cognitive Cause）
- 開発者の思い込み・見落とし
- ドキュメント不足による誤解、暗黙の前提など

### 5. 再発防止策（Prevention Plan）
- 短期対策（即座に実施可能）
- 中期対策（1-2週間で実施）
- 長期対策（設計・プロセス改善）

## 出力フォーマット

```markdown
# Postmortem Report

## 基本情報
- **日時**: YYYY-MM-DD HH:MM
- **深刻度**: Critical / High / Medium / Low
- **影響範囲**: （対象ユーザー・機能）
- **ステータス**: 調査中 / 対応済み / 監視中

## バグ概要
（1-2文で概要）

## 技術原因
- **該当コード**: `ファイルパス:行番号`
- **原因**: （詳細）

## 構造原因
- （詳細）

## プロセス原因
- （詳細）

## 認知原因
- （詳細）

## 再発防止策
### 短期（即時）
- [ ] （対策1）

### 中期（1-2週間）
- [ ] （対策2）

### 長期（設計改善）
- [ ] （対策3）

## タイムライン
| 時刻 | イベント |
|------|---------|
| HH:MM | 発見 |
| HH:MM | 対応開始 |
| HH:MM | 修正完了 |
```

## 注意事項
- 推測と事実を明確に分離する
- 個人を責めない（仕組みの問題として分析）
- 出力はNotionのページにそのままペースト可能な形式にする
